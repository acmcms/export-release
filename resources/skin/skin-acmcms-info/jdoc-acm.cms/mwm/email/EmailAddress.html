Class EmailAddress

  • java.lang.Object
    • mwm.email.EmailAddress
  • All Implemented Interfaces:
    java.io.Serializable

    public final class EmailAddress
extends java.lang.Object
implements java.io.Serializable
    This class models an immutable email address object. The immutablility makes sure that once created, the address object cannot change. The modifiers in this class works as the modifiers in the String class - they all return new objects that reflects the change - the original object is unchanged. This class also provides a static method for email address verification and one for parsing an address from a string.

    A valid name object according to the RFC 822 is a String that does not contain the characters 0 - 31, 34 or 127 which is control characters, delete and qoute. This class will ruthlessly accept ANY strings as valid names and it is up to the implementors and users of this class to make sure that the name string get encoded or left out before attempting to use it in a protocol specific manner.

    A valid address string cannot - like the name string - contain the characters 0 - 31, 34 or 127. It also cannot contain spaces. An address must also contain a "@" character which cannot start or end the address string.

    Upon creation of the address object an InvalidFormatException will be thrown if the address does not conform to the above rules.

      • parseAddress

        public static EmailAddress parseAddress(java.lang.String input)
                                 throws InvalidFormatException
        This method tries to parse a string into an email address object, according to the RFC 822 address rules. If the parsing fails the method will throw an invalid format exception.

        The parser will attempt to read a single address line wich may contain escape characters and foldings but not encodings. Thus the following lines will all be valid addresses:

        - "Lars J. Nilsson" <myaddress@address.com>
        - Lars \"J.\" Nilsson myaddress@address.com
        - "Lars \"J.\" Nilsson" <myaddress@address.com>

        The parsing mechanism attempts to unfold the given value if it contains line breaks. However, this class is considered be within a mail protocoll context (protocoll server space), meaning that it does not support linear white space within address domain tokens. That limitation also rules out folding within tokens. Thus the following string is regarded as valid...

        - "Lars J. Nilsson"
          myaddress@address.com

        ... while the following two examples will fail to parse correctly due to the linear white space. They are considered invalid:

        - "Lars J. Nilsson" myaddress @ address.com

        - "Lars J. Nilsson" myaddress@
          address.com

        Most email constructors will honor the recomendation from RFC 822 that folding is limited to "higher-level syntactic breaks" and that no linear white space shall occur between word tokens and their dots or "at"-signs when dealing with a protocoll server. For addresses this means that they will very seldon be folded other than between addresses in a comma separated list. Still it is recomended that the unfolding of addresses is done by the mail reading mechanism during in or output, and that the same mechanism also checks for linear white space.
